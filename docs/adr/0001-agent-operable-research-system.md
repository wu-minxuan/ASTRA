# ADR 0001：Agent 可操作的研究与策略实验系统架构原则

## 状态

已接受

## 背景

ASTRA 的长期目标不是构建一个单点选股工具，也不是把多个 AI 功能松散拼接在一起。ASTRA 的目标是成为一个自适应策略研究与测试 Agent 系统，支持从主题发现、股票池构建、证据研究、因子研究、策略构建、回测、模拟盘到策略 A/B 实验和持续复盘的研究闭环。

Phase 1 当前聚焦 Theme-to-Pool Research Funnel，但 Phase 1 的数据源、模型接口、模块边界和验证策略会影响后续回测、模拟盘和策略自迭代能力。因此，在进入 P1-T04 股票数据 API 选型与数据源规格前，需要先明确系统级架构原则，避免仅围绕当前召回任务做短视设计。

用户明确希望 ASTRA 最终每个交易和研究模块都是 AI Native，并且系统搭建完成后，可以让 Agent 操作不同模块、制定不同策略，并根据回测和模拟盘表现进行自迭代。

## 决策

ASTRA 采用以下长期架构原则：

```text
确定性内核 + AI 控制面 + 可追踪实验记忆
```

AI Native 在 ASTRA 中不等于每个计算步骤都由大模型完成，而是指每个研究、数据、策略、回测和模拟盘模块都必须能被 Agent 以结构化方式理解、调用、观察、评估和复盘。

### Agent 可操作模块

ASTRA 的核心模块应逐步设计为 Agent-operable module。每个模块至少应明确：

- 模块职责和非职责。
- 输入 schema。
- 输出 schema。
- 能力描述，即 Agent 可以用它解决什么问题。
- 依赖的数据源、模型规格、运行环境和权限边界。
- 错误类型、失败降级和不可用状态。
- 输出验证规则和质量评估方式。
- 成本、时延和风险边界。
- 是否需要人类审批。
- run artifact 记录要求。

长期模块应包括但不限于：

- 主题发现与 Theme-to-Pool。
- 股票池构建与更新。
- 基本面、财报、公告、新闻和研报分析。
- 因子与特征研究。
- 策略定义与参数空间。
- 回测引擎。
- 回测评估与失败诊断。
- 模拟盘执行与跟踪。
- 策略 A/B 实验。
- 策略注册表、实验记忆和复盘记录。

### 确定性内核

以下能力应优先保持确定性、可复现和可测试：

- 数据源 adapter 和字段映射。
- 股票主数据、交易日历、行情、复权和财务数据处理。
- 因子计算和特征生成。
- 回测撮合、交易成本、滑点、停牌、涨跌停和持仓约束处理。
- 收益、回撤、波动、换手、胜率、暴露和风险指标计算。
- 实验对比、样本切分和统计检验。

大模型不应替代这些确定性计算，也不应成为自动化测试稳定性的必要依赖。

### AI 控制面

大模型和 Agent 主要用于高价值认知环节：

- 将模糊研究问题转化为结构化任务。
- 主题理解、概念扩展和候选解释。
- 证据归纳、冲突识别和风险总结。
- 生成策略假设和参数搜索方向。
- 解释回测结果、识别失败原因和提出下一轮实验。
- 生成研究报告、复盘报告和实验摘要。

模型调用必须通过统一 model client interface，并使用结构化输出 schema。业务模块不得直接依赖具体模型供应商 SDK、HTTP 响应结构或自由文本输出。

### 可追踪实验记忆

ASTRA 应把 run artifact 作为一等对象。一次研究、一次策略生成、一次回测、一次模拟盘调整或一次 A/B 实验，都应尽量记录：

- 目标和触发来源。
- 输入参数和约束。
- 数据源、数据版本、时间点和快照语义。
- 模型规格、提示词版本和结构化输出 schema。
- 模块版本和代码版本。
- 输出结果、评分、证据和风险。
- 验证结果、失败原因和警告。
- Agent 的下一步建议或待验证假设。
- 人类审批记录，如果涉及策略晋级或模拟盘/实盘边界。

这些记录用于支持后续复盘、自迭代、策略对比和长期方法论沉淀。

### 自迭代边界

ASTRA 可以支持 Agent 基于回测和模拟盘表现提出策略迭代建议，但必须遵守以下边界：

- Agent 可以提出研究计划、策略假设、参数调整和下一轮实验。
- Agent 可以自动运行不涉及真实交易的研究、回测和离线评估任务。
- Agent 不能在没有人类审批的情况下接入真实交易或自动下单。
- 从研究结果晋级到模拟盘、从模拟盘晋级到真实交易，都必须有显式审批点。
- 研究输出、报告和模型排序不得包含确定性买入、卖出、持仓、目标价、仓位建议或交易时机指令。

## 理由

该架构原则有以下好处：

- 让 ASTRA 从早期就围绕研究闭环建设，而不是堆叠孤立工具。
- 保证每个模块未来都能被 Agent 编排、调用和评估。
- 保留金融系统中必须确定、可复现、可测试的计算内核。
- 将大模型用于解释、归纳、规划和诊断等高价值环节，控制成本和不确定性。
- 为回测、模拟盘、策略 A/B 实验和长期自迭代提供统一记录基础。
- 为 Phase 1 的数据源、模型接口和模块合同提供长期设计约束。

## 影响

从本 ADR 接受后，后续模块设计应尽量增加 Agent 可操作性描述，包括输入、输出、能力边界、错误形态、验证方式和 run artifact 记录要求。

P1-T04 股票数据 API 选型与数据源规格必须考虑长期系统需要，而不仅是当前候选召回。数据源规格至少应说明：

- Phase 1 最小接入字段。
- 未来是否能扩展到股票主数据、交易日历、行情、复权、财务、公告、新闻或研报。
- 外部原始字段如何通过 adapter 转换为 ASTRA 内部合同。
- 数据时间点、快照和 `as_of` 语义。
- 许可、使用边界、限流和失败降级方式。
- 自动化测试如何使用 fixture 或 mock，避免依赖真实网络。

后续关于模型供应商、数据源、回测引擎、模拟盘和策略注册表的关键选择，应继续用 ADR 记录。

## 替代方案

### 只构建固定业务流程

可以只做一个从主题到股票池的固定 Web 流程。这种方案更快，但后续难以支持 Agent 编排、策略实验和自迭代。

### 所有模块都由大模型驱动

可以让大模型直接处理数据、生成策略、解释结果和给出下一步动作。这种方案看起来更 AI Native，但会牺牲可复现性、测试稳定性、金融计算可靠性和风险控制。

### 先做完整平台架构再实现 Phase 1

可以先设计完整数据仓库、Agent 调度、回测平台和实验平台。这种方案更完整，但会违背 ASTRA 阶段性 MVP 原则，容易过度设计。

## 重新评估条件

出现以下情况时，应重新评估本 ADR：

- ASTRA 的目标从个人研究和 Agent 工程实践转为生产级交易系统。
- 系统开始接入真实交易或自动下单能力。
- 回测、模拟盘或策略实验需要更严格的审计、权限和合规边界。
- 当前 run artifact 记录方式无法支持复盘、自迭代或实验对比。
- 模型调用、数据源或 Agent 调度架构发生跨阶段重大变化。
