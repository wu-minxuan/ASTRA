# Phase 1：主题到股票池研究漏斗

Phase 1 的目标是完成 ASTRA 的第一个业务 MVP：输入一个主题或市场热点，输出可解释的 A 股主题股票池与研究报告。

这个阶段采用 Theme-to-Pool Research Funnel 作为实现路径。它只是 ASTRA 的第一阶段 MVP，不是 ASTRA 的完整定义。

## 阶段目标

Phase 1 需要建立一条可运行、可验证、可复盘的研究闭环：

- 用户输入一个主题或热点，例如“低空经济”。
- 系统解析主题，并召回相关 A 股候选公司。
- 系统补充候选公司的基础证据，包括概念、行业、基本面摘要、财务摘要和文本资料摘要。
- 系统通过召回、粗排、精排的漏斗流程控制成本和质量。
- 系统输出股票池、排序理由、证据链、风险提示和研究报告。
- 系统提供前端页面和自动化测试，验证最小端到端流程。

## 阶段范围

Phase 1 包括：

- 定义主题研究请求、候选股票、证据项、评分结果和研究报告的数据结构。
- 建立主题解析与查询归一化能力。
- 建立候选股票召回模块。
- 建立候选股票证据补全模块。
- 建立粗排模块，用低成本规则完成初步筛选。
- 建立精排模块，用更高成本的分析逻辑生成最终排序和解释。
- 建立研究报告生成模块。
- 建立后端 API，让前端可以提交主题并获取研究结果。
- 建立最小前端页面，用于输入主题、查看股票池和研究报告。
- 建立测试样例、固定数据和自动化验收，确保流程可复现。

## 非范围

Phase 1 不包括：

- 覆盖港股、美股、ETF 或其他资产。
- 实现量化因子研究、策略回测、模拟盘或交易系统。
- 接入真实交易或生成自动下单指令。
- 输出确定性买入、卖出或持仓建议。
- 构建完整数据仓库、特征平台或生产级数据同步系统。
- 构建复杂 Agent 调度框架。
- 部署或优化自有模型服务。
- 构建完整 Web UI 或设计系统。

## 漏斗流程

Phase 1 的研究流程采用漏斗结构：

```text
主题输入
  -> 主题解析
  -> 候选股票召回
  -> 证据补全
  -> 粗排
  -> 精排
  -> 股票池与研究报告
```

漏斗设计原则：

- 召回阶段追求覆盖，允许噪声。
- 粗排阶段追求低成本过滤，优先使用结构化规则和可解释指标。
- 精排阶段追求研究质量，才使用更高成本的分析能力。
- 每个阶段都应保留输入、输出和关键理由，便于复盘。
- 大模型能力优先用于主题理解、复杂文本归纳、精排解释和报告生成，不用于无差别处理全量数据。

## 数据原则

Phase 1 的验收应优先依赖固定样例数据，保证流程可复现。

真实行情、财务、公告、研报、新闻或概念数据源可以在 Phase 1 中逐步接入，但接入前必须明确：

- 数据来源
- 数据字段
- 更新时间
- 许可与使用边界
- 失败降级方式
- 测试替代数据

外部数据源不应成为 Phase 1 最小验收的唯一依赖。

## 输出要求

Phase 1 输出的股票池应至少包含：

- 股票代码
- 股票名称
- 召回来源
- 排名或评分
- 入选理由
- 关键证据
- 主要风险

Phase 1 输出的研究报告应至少包含：

- 主题概述
- 主题与候选公司的关系说明
- 股票池汇总
- 重点公司解释
- 风险提示
- 数据与证据边界说明

输出必须区分事实、推断和假设，不能伪装成确定性投资建议。

## 验证要求

Phase 1 的验证类型固定为：

- 静态检查
- 单元测试
- 集成测试
- 端到端测试

Phase 1 的端到端测试应使用 Playwright 验证：

- 用户可以在前端输入主题。
- 页面可以展示股票池结果。
- 页面可以展示研究报告摘要。
- 当后端返回错误时，页面可以展示错误状态。

## 验收标准

Phase 1 完成时，必须满足：

- 用户可以从前端输入一个主题并触发研究流程。
- 后端可以返回结构化股票池和研究报告。
- 漏斗各阶段有清晰输入、输出和可测试边界。
- 至少一个固定样例主题可以稳定复现完整结果。
- 输出包含解释、证据和风险提示。
- `make check` 通过。
- 端到端测试覆盖最小主题研究流程。
- 研究输出不包含确定性投资建议或交易指令。

## 进入下一阶段的条件

只有在 Phase 1 验收标准满足后，项目才进入下一阶段。

进入下一阶段前，应已经明确：

- 真实数据源接入策略。
- 主题股票池质量评估方法。
- 后续是否优先扩展数据、报告质量、因子研究或回测能力。
