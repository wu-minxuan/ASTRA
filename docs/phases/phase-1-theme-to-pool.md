# Phase 1：主题到股票池研究漏斗

Phase 1 的目标是完成 ASTRA 的第一个业务 MVP：输入一个主题或市场热点，输出可解释的 A 股主题股票池与研究报告。

这个阶段采用 Theme-to-Pool Research Funnel 作为实现路径。它只是 ASTRA 的第一阶段 MVP，不是 ASTRA 的完整定义。

## 阶段目标

Phase 1 需要建立一条可运行、可验证、可复盘的研究闭环：

- 用户输入一个主题或热点，例如“低空经济”。
- 系统解析主题，并召回相关 A 股候选公司。
- 系统在固定样例数据之外，明确并实现一个真实股票数据 API 的最小接入路径。
- 系统补充候选公司的基础证据，包括概念、行业、基本面摘要、财务摘要和文本资料摘要。
- 系统通过召回、证据补全、模型粗排、模型精排的漏斗流程控制成本和质量。
- 系统输出股票池、排序理由、证据链、风险提示和研究报告。
- 系统提供前端页面和自动化测试，验证最小端到端流程。

## 阶段范围

Phase 1 包括：

- 定义主题研究请求、候选股票、证据项、评分结果和研究报告的数据结构。
- 建立主题解析与查询归一化能力。
- 建立候选股票召回模块。
- 完成真实股票数据 API 的选型、数据源规格和许可边界说明。
- 实现股票数据源 adapter，将真实 API 原始数据转换为 ASTRA 内部研究合同。
- 实现真实数据源候选召回接入，并保留固定样例数据作为测试替代和失败降级路径。
- 建立候选股票证据补全模块。
- 建立模型粗排模块，用低成本模型完成初步筛选、评分和过滤理由生成。
- 建立模型精排模块，用更高质量模型完成最终排序、解释、风险判断和不确定性说明。
- 建立模型调用抽象、模型规格、结构化输出 schema 和测试用 fake/mock model client。
- 建立研究报告生成模块。
- 建立后端 API，让前端可以提交主题并获取研究结果。
- 建立最小前端页面，用于输入主题、查看股票池和研究报告。
- 建立测试样例、固定数据和自动化验收，确保流程可复现。

## 非范围

Phase 1 不包括：

- 覆盖港股、美股、ETF 或其他资产。
- 实现量化因子研究、策略回测、模拟盘或交易系统。
- 接入真实交易或生成自动下单指令。
- 输出确定性买入、卖出或持仓建议。
- 构建完整数据仓库、特征平台或生产级数据同步系统。
- 构建复杂 Agent 调度框架。
- 部署或优化自有模型服务。
- 构建完整 Web UI 或设计系统。

## 漏斗流程

Phase 1 的研究流程采用漏斗结构：

```text
主题输入
  -> 主题解析
  -> 候选股票召回
  -> 证据补全
  -> 模型粗排
  -> 模型精排
  -> 股票池与研究报告
```

漏斗设计原则：

- 召回阶段追求覆盖，允许噪声，优先使用规则匹配、数据源标签和低成本可解释逻辑。
- 证据补全阶段追求证据结构化、来源保留和输入压缩，为后续模型判断提供可控上下文。
- 粗排阶段必须通过模型接口完成低成本评分、过滤和初步理由生成，优先使用较低成本、快速、结构化输出的模型规格。
- 精排阶段必须通过模型接口完成最终排序、解释、风险判断和不确定性说明，可使用更高质量、更长上下文或更强推理能力的模型规格。
- 报告生成必须基于前序结构化证据和模型排序结果，不能脱离证据自由生成确定性荐股结论。
- 每个阶段都应保留输入、输出和关键理由，便于复盘。
- 大模型能力用于主题理解、复杂文本归纳、粗排、精排解释和报告生成，不用于无差别处理全量原始数据。

## 模型原则

Phase 1 的 Theme-to-Pool 是完整研究漏斗，不是单纯候选召回工具。

模型接入必须遵守以下原则：

- 粗排和精排必须通过统一 model client interface 调用，不能把模型供应商 API 散落在业务逻辑中。
- 粗排和精排可以使用不同模型规格，分别记录为 `coarse_rank_model_spec` 和 `deep_rank_model_spec`。
- 粗排模型输入应是压缩后的候选摘要和关键证据，输出必须是结构化评分、保留/过滤判断、理由和风险摘要。
- 精排模型输入应是候选证据包和粗排结果，输出必须是最终分数、排序理由、关键证据引用、主要风险和不确定性。
- 自动化测试不得依赖真实模型输出的稳定性，必须使用 fake/mock model client 和固定结构化响应。
- 模型输出必须经过 schema 校验和安全边界检查，不得包含买入、卖出、持仓、目标价或交易指令。

## 数据原则

Phase 1 采用固定样例数据和真实股票数据 API 双轨数据策略。

固定样例数据用于保证流程可复现、测试稳定和失败降级。真实股票数据 API 用于验证 Theme-to-Pool MVP 具备最小真实数据接入能力。

Phase 1 必须完成一个真实股票数据 API 的最小接入。最小接入范围包括：

- A 股股票基础信息，包括代码、名称、交易所、上市状态和行业分类中的可得字段。
- 主题、概念、板块或行业成分数据，用于把主题输入映射到候选公司。
- 数据源 adapter，将外部 API 原始字段转换为 ASTRA 内部研究合同。
- 失败降级路径，当外部 API 不可用、限流或字段缺失时，可以退回固定样例数据完成可复现测试。

真实股票数据 API 接入前必须明确：

- 数据来源
- 数据字段
- 更新时间
- 许可与使用边界
- 失败降级方式
- 测试替代数据

外部数据源不应成为 Phase 1 自动化验收的唯一依赖。固定样例数据必须继续作为 CI 和本地测试的稳定替代数据。

Phase 1 不要求接入完整行情、完整财务、公告、研报或新闻数据。上述数据可以在 Phase 1 内后续扩展，但不属于真实股票数据 API 最小接入的必要范围。

## 输出要求

Phase 1 输出的股票池应至少包含：

- 股票代码
- 股票名称
- 召回来源
- 排名或评分
- 入选理由
- 关键证据
- 主要风险

Phase 1 输出的研究报告应至少包含：

- 主题概述
- 主题与候选公司的关系说明
- 股票池汇总
- 重点公司解释
- 风险提示
- 数据与证据边界说明

输出必须区分事实、推断和假设，不能伪装成确定性投资建议。

## 验证要求

Phase 1 的验证类型固定为：

- 静态检查
- 单元测试
- 集成测试
- 端到端测试

Phase 1 的端到端测试应使用 Playwright 验证：

- 用户可以在前端输入主题。
- 页面可以展示股票池结果。
- 页面可以展示研究报告摘要。
- 当后端返回错误时，页面可以展示错误状态。

## 验收标准

Phase 1 完成时，必须满足：

- 用户可以从前端输入一个主题并触发研究流程。
- 后端可以返回结构化股票池和研究报告。
- 漏斗各阶段有清晰输入、输出和可测试边界。
- 至少一个固定样例主题可以稳定复现完整结果。
- 至少一个真实股票数据 API 已完成选型、规格记录、adapter 实现和最小候选召回接入。
- 真实数据源不可用时，系统可以降级到固定样例数据完成可复现流程。
- 粗排和精排均通过模型接口完成，并具备不同模型规格的配置边界。
- 模型输出经过结构化 schema 校验，并覆盖禁止交易指令的自动化测试。
- 输出包含解释、证据和风险提示。
- `make check` 通过。
- 端到端测试覆盖最小主题研究流程。
- 研究输出不包含确定性投资建议或交易指令。

## 进入下一阶段的条件

只有在 Phase 1 验收标准满足后，项目才进入下一阶段。

进入下一阶段前，应已经明确：

- 真实数据源接入策略已经在 Phase 1 中完成最小实现，下一阶段只扩展数据深度或覆盖范围。
- 主题股票池质量评估方法。
- 后续是否优先扩展数据、报告质量、因子研究或回测能力。
