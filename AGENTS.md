# AGENTS.md

这个仓库用于 ASTRA 项目的长时运行 AI Coding 工作流。目标不是尽快产出代码，而是让每一轮讨论、规划和实现都能被后续会话无猜测地继承。

ASTRA 是一个自适应策略研究与测试 Agent 系统，支持主题发现、股票池构建、基本面分析、研报分析、量化因子研究、策略回测、模拟盘和策略 A/B 实验。

## 开工流程

进行任何项目状态变更前，先做这些事：

1. 确认当前任务处于讨论模式、规划模式、执行模式还是审查模式。
2. 读取 `AGENTS.md`。
3. 读取当前 Phase 文档。
4. 读取当前任务规格，如果存在。

如果必需文档尚不存在，只能继续讨论或起草内容，不能自行创建或补写文件。

## 工作规则

- 默认只讨论；任何会改变项目状态的操作，都必须先得到用户明确授权。
- 一次只做一个进行中的任务，即 WIP=1。
- 不要扩大到未经授权的任务范围。
- 不要因为内容已经写出或代码已经修改，就把任务标记为完成。
- 不要悄悄改弱验证规则或验收标准。
- 优先依赖仓库中的持久化文件，而不是聊天记录。
- 当规则、任务范围或用户意图不清楚时，先停下并向用户确认。

## 真实性与透明度规则

所有任务执行过程中，Agent 必须对用户真实、完整、透明，不保留、不隐瞒。

- 必须明确区分“已设计”“已实现”“已接入主流程”“已用 mock/fake/fixture 验证”“已用真实外部服务验证”“已提交”“已推送”等不同状态。
- 任何测试、验证或演示如果使用 mock、fake、fixture、本地替代、缓存、fallback、跳过项或降级路径，必须主动说明，不能让用户误以为它覆盖了真实外部依赖。
- 涉及真实互联网、真实股票数据 API、模型 API、数据库、券商接口、浏览器端到端等外部能力时，必须说明是否真的访问、访问了什么接口、结果如何，以及失败、警告或限制是什么。
- 不得把“代码中存在 provider、adapter 或 client”表述成“真实数据源已经验证可用”，除非已经运行对应真实调用并说明验证证据。
- 对失败、跳过、flaky、网络错误、上游接口变化、权限或许可限制、字段缺失、fallback、degradation 和未解决风险，必须如实报告。
- 基于推断、假设或聊天记忆的结论必须明确标记；优先用仓库文件、命令输出和测试结果支撑结论。
- 当发现之前表述不准确、不完整或可能误导用户时，必须主动纠正并补充到当前任务记录或相关文档中。

## 项目状态变更

项目状态变更包括但不限于：

- 创建、修改、删除、移动或格式化文件和目录
- 安装、升级或删除依赖
- 初始化项目、生成代码或生成配置
- 运行会写入文件、缓存、数据库或其他持久状态的命令
- 修改 Git 状态，例如暂存、提交、回滚、变基或合并

## 验证规则

任务完成必须提供与任务影响范围匹配的验证证据。

ASTRA 项目的验证类型固定为：

- 静态检查
- 单元测试
- 集成测试
- 端到端测试

本项目测试选型：

- `pytest` 作为通用测试入口，承载单元测试、集成测试和非浏览器端到端测试。
- `Playwright` 作为浏览器端到端测试工具。
- Codex Browser 插件用于补充真实浏览器交互验证，但不替代 Playwright 端到端测试。

## 完成定义

一个执行任务只有在以下条件都满足时才算完成：

- 目标行为已经实现
- 修改范围没有超出授权
- 已提供与任务影响范围匹配的验证证据，或明确说明无法验证的原因
- 失败、跳过、风险和后续事项已经说明
- 仓库仍处于可继续工作的状态

## 收尾

结束一次执行任务前，必须向用户报告：

1. 实际完成了什么。
2. 修改了哪些文件。
3. 运行了哪些验证。
4. 验证结果如何。
5. 是否存在未解决的风险或 blocker。
6. 是否需要用户批准下一步状态记录或文档更新。

未经用户明确要求，不得自动提交代码。
